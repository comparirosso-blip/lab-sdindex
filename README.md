# Lab_SDIndex

A research repository for SDIndex protocol development, structured field observation, and future Graph RAG alignment.

## What This Repository Is

Lab_SDIndex is a research-oriented repository for developing, documenting, and maintaining the **SDIndex** system.

This is not a casual note archive, a travel diary collection, or a generic JSON storage folder.

It is a working research repository for:

- structured sensory field observation
- schema-based record standardization
- canonical dataset formation
- QC and audit traceability
- future Graph RAG / knowledge graph interoperability

The repository is designed around one core principle:

**human observation first, structured consistency second, graph connection later**

## Core Research Logic

The SDIndex framework is based on the view that many meaningful spatial and experiential qualities cannot be adequately described by conventional tourism language, star ratings, or generic metadata.

This repository exists to support a more rigorous alternative:
a structured method for recording, reviewing, and stabilizing human sensory observations.

Its foundation rests on three core capacities:

- **Observation**  
  Direct, embodied noticing of space, atmosphere, contrast, rhythm, permeability, compression, and affective response.

- **Schema Thinking**  
  Translating rich subjective experience into a repeatable and structured recording system.

- **Consistency**  
  Applying shared logic across records so they can later be compared, audited, and computationally connected.

## Repository Goals

This repository supports five major goals:

1. maintain a stable SDIndex protocol environment
2. store field observations in structured form
3. distinguish raw, canonical, and analytical layers of data
4. preserve QC and revision traceability
5. prepare selected records for future Graph RAG compatibility

Directory Meaning
README.md

The entry point of the repository.
Explains the purpose, logic, and positioning of Lab_SDIndex.

REPO_STRUCTURE.md

Documents the repository architecture and explains why each folder exists.

DATA_LIFECYCLE.md

Defines how records move from raw observation to reviewed, canonical, and analysis-ready states.

data/raw/

Stores original field records before full normalization, validation, or canonicalization.

These may include:

early structured observations

initial JSON entries

partial records

records pending QC review

data/canonical_records/

Stores finalized records that have passed sufficient review and conform to current protocol standards.

These are the closest thing to the repository’s stable source set.

Not every raw record should become canonical.

data/analysis/

Stores interpretive, comparative, or derived materials based on canonical records.

Examples include:

dimension consistency analysis

cross-site comparison

thematic clustering

future graph mapping notes

benchmark subsets

data/qc_audit_notes/

Stores review notes, correction logs, ambiguity flags, and methodological audit trails.

This folder is critical because SDIndex is not only about storing observations.
It is also about preserving the reasoning behind why a record was revised, accepted, questioned, or rejected.

schema/

Stores formal schema definitions, controlled vocabularies, field rules, and validation logic.

This folder defines the structural boundaries of the repository.

docs/

Stores protocol explanations, methodology notes, conceptual essays, and implementation guidance.

This is where the repo explains itself.

tools/

Stores scripts, helper utilities, converters, validation tools, and future automation logic.

This folder should remain secondary to the protocol itself.
Tools serve the method, not the other way around.

research_log/

Stores ongoing research notes, design decisions, methodological reflections, naming decisions, and evolution history.

This folder preserves the intellectual development of the system.

Data Lifecycle Philosophy

A key principle of this repository is that not all records have the same epistemic status.

For that reason, the data layer is intentionally separated into:

raw observations

canonical records

analysis outputs

QC / audit notes

This separation prevents premature flattening.

A raw observation is not automatically reliable enough for downstream inference.
An analysis output is not the same thing as a source observation.
A QC note is not noise; it is part of the methodological integrity of the system.

Relation to Graph RAG

This repository is being built with future Graph RAG alignment in mind, but Graph RAG is not the starting point.

The correct order is:

protocol clarity

record accumulation

consistency review

canonical stabilization

graph mapping

retrieval and reasoning use cases

Possible future Graph RAG questions may include:

Which Kyoto sites show high Ve under low-crowd conditions?

Which records reflect strong Vt but low relational permeability?

Which canonical observations best represent compressed urban transcendence?

Which sites repeatedly generate similar sensory vectors across seasons?

These become meaningful only after enough stable records exist.
